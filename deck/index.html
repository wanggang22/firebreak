02
the problem

Liquidation doesn't flash-crash.
It drifts.

On a stablecoin-native L1, collateral is FX and tokenized RWAs — EURC, T-bills, private credit. They don't gap down 40% in a candle. They slide: the euro moves two cents over a week, an RWA re-prices, interest accrues.

Health factor walks slowly toward the line. A liquidator bot has been waiting for hours. It takes a 5–10% penalty out of the borrower's collateral — for something a 0.94 USDC top-up would have fixed.

Nobody is watching their health factor at 3am. And handing a bot custody, or approving each transaction by hand, both defeat the purpose.

the counterfactual, on-chain

Two identical positions drift down. The unprotected one is liquidated — all 100 mEURC seized. Firebreak acts early at the trigger and the twin stays solvent — 83 mEURC kept, never liquidated. Same market, opposite outcomes.

03
the solution

A bounded authorization the borrower signs on-chain.

Trigger

< 1.20

Act only when health factor crosses this line.

Spend cap

50 USDC

Never move more than this per rescue.

Whitelist

DELEVERAGE
ROTATE · TOP-UP

Only these actions, nothing else.

The FirebreakMandate contract re-checks every bound on-chain at execution. The keeper never holds borrower funds and cannot exceed the cap, take an un-whitelisted action, or act on a healthy position.

Non-custodial and autonomous are not a tradeoff — if the bounds live in the contract.

04
how it works

Monitor → strategist → executor, in one atomic tx.

signals ┌──────────────────────────────┐ HF, prices, ───▶ │ Keeper agent (TypeScript) │ swap quotes │ monitor → strategist → exec │ └───────────────┬──────────────┘ │ rescue(user, plan) ▼ ┌──────────────────────────────┐ │ FirebreakMandate.sol │ bounded · conditional │ HF < trigger? plan in bounds?│ non-custodial └───────┬───────────────┬──────┘ ▼ ▼ MiniLend.sol / ScaledLend.sol MiniSwap.sol (two IPosition adapters) (USDC ↔ ERC-20)

The keeper speaks a small IPosition interface. Two adapters back it — MiniLend keeps raw amounts, ScaledLend keeps share-based collateral and index-accrued debt — and the same Mandate bytecode guards both, so "protocol-agnostic" is a tested property rather than a claim.

05
the hard part — and our differentiator

The LLM ranks. Deterministic code sizes and bounds-checks.

Deterministic core

  • Builds every allowed rescue path
  • Sizes each to restore health to target (closed form)
  • Prices each, drops any breaching the spend cap

→ a set of vetted, executable candidates

Claude ranks — only that set

  • Strict-tool enum of candidate action ids
  • Writes the borrower-facing reasoning memo
  • Out-of-set pick or API error → cheapest path executes anyway

→ never stalls, never unbounded

The model's only freedom is which safe path — never whether it's safe. Sizing is arithmetic and stays in code. 11/11 harness tests pin exactly this contract.

where the LLM earns its place — executed on-chain

Small reserve: the cheapest path (zero-cost TOP-UP) only partially fixes health (1.266, still at risk). Claude picks ROTATE — a full fix to 1.380 for 0.36 USDC — because it sees each path's projected HF. The naive cheapest-by-cost rule gets this wrong. HF 1.190 → 1.602, position rotated into higher-quality collateral.

09
why this wants Arc specifically

Three properties, each load-bearing.

01 · drifting collateral

FX and tokenized RWAs are Arc's native collateral. Slow drift is exactly what a watching agent catches and a human misses. On a chain of volatile alts you'd be too late anyway.

02 · sub-second settlement

The rescue must land before the liquidator's transaction. Rescue and liquidation are in a race — finality latency is the margin.

03 · USDC-denominated gas

Keeper cost and borrower savings are in the same unit. You can decide whether a 0.94 USDC rescue is worth sending, without modeling a volatile gas token.

Firebreak is not a generic DeFi app ported to Arc. Remove any one of these three and the product stops working.

10
not the same as existing automation

They bound when. We bound when + how much + what.

DeFi Saver · Instadapp · Summer.fiFirebreak
authorization Migrate the position into their smart wallet / proxy, or grant a broad automation executor A per-position Mandate: trigger + spend cap + action whitelist + slippage floor, signed on-chain
where it's enforced In their automation backend — you trust the bot to stay in scope In the contract, at execution — the keeper is untrusted by construction
the rescue decision Threshold triggers a preset action An LLM ranks the vetted paths and writes an auditable memo — durability over cost
chain EVM L2s built for volatile collateral that gaps Arc — where FX/RWA collateral drifts, the case these products don't target

The honest line: incumbents do store trigger params on-chain. What's new is that the *entire authority* — amount, action, slippage — is a bounded object the borrower signs, enforced by the contract, not a scope you trust an off-chain bot to respect.

11
what's next

From hackathon to production.

  • Circle wallet spend policies — the keeper already runs on an Agent Wallet; when Arc reaches mainnet its policy becomes a second, Circle-enforced bound alongside the Mandate
  • A live Arc lending venue — two independently-shaped adapters already show the IPosition abstraction holds; a third against a deployed third-party pool is integration work, not design risk
  • The funded cross-chain refill — the Unified Balance integration is verified live against Gateway and the refill policy is tested, but the funded deposit → delegate → spend has not been run on testnet yet
  • Agent identity — authorize a verified agent identity (ERC-8004 / KYA) instead of a raw keeper address, so a borrower can revoke a class of keeper
  • Multi-keeper coordination — first-writer-wins on-chain today; commit-reveal if racing becomes real
